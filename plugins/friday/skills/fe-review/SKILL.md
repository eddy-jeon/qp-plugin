# Frontend Code Review Guide

## 리뷰 관점

### 1. 컴포넌트 구조 및 책임 분리

- 하나의 컴포넌트가 너무 많은 역할을 하고 있지 않은지
- 재사용 가능한 로직이 적절히 분리되어 있는지 (custom hooks, utility 함수)
- 컴포넌트 계층 구조가 합리적인지
- 프레젠테이션과 비즈니스 로직이 분리되어 있는지

### 2. 상태 관리

- 상태가 적절한 레벨에 위치하는지 (로컬 vs 전역)
- 불필요한 상태가 없는지 (파생 가능한 값을 별도 상태로 관리하고 있지 않은지)
- 상태 업데이트 로직이 예측 가능한지
- 비동기 상태 관리가 적절한지 (loading, error, data)

### 3. 타입 안전성

- `any` 타입 사용을 최소화하고 있는지
- API 응답 타입이 정의되어 있는지
- 유니온 타입, 제네릭 등을 적절히 활용하고 있는지
- `as` 타입 단언의 남용이 없는지

### 4. 성능

- 불필요한 리렌더링이 발생하지 않는지
- `useMemo`, `useCallback`이 필요한 곳에 사용되었는지 (과도한 사용도 지적)
- 대량 데이터 렌더링 시 가상화가 적용되어 있는지
- 번들 사이즈에 영향을 주는 import가 있는지 (tree-shaking 가능 여부)
- 이미지/미디어 최적화가 되어 있는지

### 5. 접근성 (a11y)

- 시맨틱 HTML을 사용하고 있는지
- 인터랙티브 요소에 적절한 ARIA 속성이 있는지
- 키보드 네비게이션이 가능한지
- 색상 대비가 충분한지 (디자인 관련 변경 시)

### 6. 에러 처리

- API 호출 실패 시 사용자에게 적절한 피드백이 있는지
- Error Boundary가 적절히 배치되어 있는지
- 엣지 케이스 (빈 데이터, null, undefined)가 처리되어 있는지

### 7. 코드 스타일 및 일관성

- 프로젝트의 기존 패턴과 일관성이 있는지
- 네이밍 컨벤션을 따르고 있는지
- 불필요한 코드 (console.log, 주석 처리된 코드)가 없는지

## 리뷰 우선순위

1. **수정 필요**: 버그, 보안 이슈, 심각한 성능 문제
2. **권장**: 코드 품질, 유지보수성 개선
3. **잘된 점**: 좋은 패턴, 개선된 부분 인정

## 피드백 작성 원칙

- **구체적**: 문제가 있는 정확한 위치와 코드를 지적
- **건설적**: 문제만 지적하지 않고 개선 방법을 함께 제시
- **근거 있는**: 왜 변경이 필요한지 이유를 설명
- **존중하는**: 코드를 비판하되 작성자를 비판하지 않음
