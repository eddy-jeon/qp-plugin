# Research Phase - Simple Engineering

당신은 **Research Phase**를 수행합니다. 이 단계의 목표는 구현 전에 철저한 조사를 통해 **Earned Understanding(축적된 이해)**을 구축하는 것입니다.

## ⚠️ 단계 전환 규칙 (필수)

**절대 원칙**: 사용자가 명시적으로 "다음 단계로 진행", "plan으로 넘어가자", "research 완료" 등을 말하기 전까지 **절대로 Plan 단계로 넘어가지 않습니다**.

### 피드백 루프 동작

1. **사용자 피드백 수신 시**: 현재 Research 단계 내에서 피드백을 반영하여 조사를 계속합니다
2. **추가 의견/질문 시**: 해당 내용을 Research에 통합하고, 필요시 재조사합니다
3. **단계 전환 요청 시에만**: Plan 단계 진행을 안내합니다

### 금지 사항

- ❌ 사용자 피드백을 받자마자 "그러면 Plan으로 넘어가겠습니다" 금지
- ❌ Research 산출물 생성 후 자동으로 Plan 명령어 실행 금지
- ❌ "다음 단계를 진행할까요?" 라고 먼저 제안하는 것 금지

### 허용 사항

- ✅ Research가 충분히 완료되었다고 판단되면 **안내만** 제공
- ✅ 사용자가 명시적으로 요청하면 Plan 단계로 전환

## 작업: $ARGUMENTS.task

## 수행 절차

### 1. 상태 초기화

먼저 상태 디렉토리를 확인하고 초기화합니다:

```bash
mkdir -p .claude/simple-engineering/{research,specs,checkpoints}
```

고유 ID를 생성합니다 (예: `research-$(date +%Y%m%d-%H%M%S)`).

### 2. 체계적 조사

다음 영역을 순차적으로 조사합니다:

#### 2.1 현재 상태 파악

- 관련 코드/파일 탐색
- 기존 구현 분석
- 의존성 파악

#### 2.2 문제/요구사항 정의

- 해결해야 할 핵심 문제 식별
- 제약 조건 파악
- 성공 기준 정의

#### 2.3 요구사항 심층 질문

**중요**: 사용자의 요구사항을 완전하게 이해하기 위해 다음 원칙을 따릅니다.

##### 객관식 질문 사용

요구사항을 명확하게 만들기 위해 **가능한 객관식으로 질문**합니다:

```
<checkpoint type="clarification" id="cp-XXX">
## 요구사항 확인

**질문**: [질문 내용]

**선택지**:
1. [Option A] - [설명]
2. [Option B] - [설명]
3. [Option C] - [설명]
4. 기타 (직접 입력)

---
번호로 선택해 주세요.
</checkpoint>
```

##### 3단계 깊이의 질문

각 요구사항에 대해 최소 3단계 깊이로 질문을 수행합니다:

| 단계 | 초점 | 질문 예시 |
|------|------|-----------|
| **1단계: 큰 그림** | 전체 목표, 범위 | "이 기능의 최종 목표는 무엇인가요?", "어떤 사용자가 이 기능을 사용하나요?" |
| **2단계: 세부사항** | 구체적인 동작, 조건 | "버튼 클릭 시 어떤 동작이 실행되어야 하나요?", "데이터는 어떤 형식으로 저장되나요?" |
| **3단계: 엣지케이스** | 예외 상황, 제약사항 | "네트워크 오류 시 어떻게 처리하나요?", "동시에 여러 요청이 올 경우 어떻게 되나요?" |

##### 숨겨진 요구사항 발굴

사용자가 미처 생각하지 못한 부분을 질문하여 완전한 요구사항을 도출합니다:

- **보안**: 인증/인가 요구사항, 민감 데이터 처리
- **성능**: 응답 시간, 동시 사용자 수, 데이터 크기 제한
- **호환성**: 브라우저/디바이스 지원, API 버전 호환성
- **접근성**: 장애인 접근성, 다국어 지원
- **유지보수**: 로깅, 모니터링, 롤백 전략

```
<checkpoint type="hidden-requirement" id="cp-XXX">
## 숨겨진 요구사항 확인

다음 항목 중 고려해야 할 사항이 있나요?

- [ ] 인증/권한 체크가 필요한가요?
- [ ] 동시 요청 제한이 있나요?
- [ ] 특정 브라우저/환경 지원이 필요한가요?
- [ ] 에러 발생 시 알림이 필요한가요?
- [ ] 기타: ___

---
해당하는 항목을 알려주세요.
</checkpoint>
```

#### 2.4 가정 도출

**중요**: 조사 중 도출된 모든 가정을 명시적으로 기록합니다.

가정을 발견할 때마다 다음 형식으로 체크포인트를 생성합니다:

```
<checkpoint type="assumption" id="cp-XXX">
## 검증 필요

**가정**: [가정 내용]
**근거**: [이 가정을 하게 된 근거]
**검증 요청**: [사용자에게 확인받고 싶은 질문]

---
이 가정이 맞는지 확인해 주세요. `continue`로 진행하거나 수정 사항을 알려주세요.
</checkpoint>
```

### 3. 복잡성 분석

조사 결과를 바탕으로 복잡성을 분석합니다:

#### Essential Complexity (본질적 복잡성)

- 문제 도메인에서 피할 수 없는 복잡성
- 비즈니스 로직의 고유한 복잡성

#### Accidental Complexity (부수적 복잡성)

- 기술적 선택으로 인해 발생한 복잡성
- 제거하거나 단순화할 수 있는 부분

### 4. 산출물 생성

조사가 완료되면 `.claude/simple-engineering/research/{task-id}.md`에 다음 형식으로 저장합니다:

```markdown
# Research: [Task Name]

**ID**: research-YYYYMMDD-HHMMSS
**Date**: [Date]
**Status**: completed | in-progress

## Executive Summary

[2-3문장 요약]

## Problem Statement

[해결해야 할 문제]

## Current State Analysis

[현재 상태 분석 결과]

## Assumptions (Verified)

- [ ] Assumption 1: [내용] - verified/pending/rejected
- [ ] Assumption 2: [내용] - verified/pending/rejected

## Complexity Analysis

### Essential

- [본질적 복잡성 항목]

### Accidental

- [부수적 복잡성 항목]

## Key Findings

[핵심 발견 사항]

## Recommendations

[다음 단계 권장 사항]

## Checkpoints Log

- cp-001: [가정] - [상태]
- cp-002: [결정] - [상태]
```

### 5. 산출물 검토 및 피드백 요청

Research 산출물을 생성한 후, **반드시 사용자와 함께 검토**합니다.

#### 5.1 산출물 요약 제시

산출물의 핵심 내용을 구조화하여 제시합니다:

```
---
📋 Research Phase 산출물 생성 완료

산출물: .claude/simple-engineering/research/{task-id}.md
---

## 📊 조사 결과 요약

### 문제 정의
[Problem Statement 핵심 1-2문장]

### 현재 상태
[Current State의 핵심 발견 사항]

### 가정 상태
| 가정 | 상태 | 비고 |
|------|------|------|
| [가정1] | ✅ 검증됨 | [근거] |
| [가정2] | ⏳ 미검증 | [확인 필요 사항] |
| [가정3] | ❌ 기각됨 | [이유] |

### 복잡성 분석
- **Essential**: [핵심 본질적 복잡성]
- **Accidental**: [제거 가능한 부수적 복잡성]
```

#### 5.2 검토 요청 체크리스트

사용자에게 다음 항목을 **명시적으로 질문**합니다:

```
---
## 🔍 검토 요청

다음 항목을 확인해 주세요:

### 1. 문제 정의 검토
> "[Problem Statement 내용]"
- 이 문제 정의가 정확한가요?
- 누락된 요구사항이 있나요?

### 2. 가정 검토
> 미검증 가정 [N]개가 있습니다:
> - [미검증 가정 1]
> - [미검증 가정 2]
- 이 가정들이 맞는지 확인해 주세요
- 추가로 검증이 필요한 부분이 있나요?

### 3. 범위 검토
> 조사 범위: [조사한 영역 요약]
- 조사하지 못한 영역이 있나요?
- 더 깊이 파야 할 부분이 있나요?

### 4. 놓친 부분 확인
- 제가 고려하지 못한 제약 조건이 있나요?
- 팀/조직 차원의 컨텍스트가 더 있나요?
---
```

#### 5.3 피드백 대기

```
💬 위 내용을 검토하시고 피드백을 주세요.
   수정/보완이 필요하면 말씀해 주시고,
   Plan 단계로 넘어가려면 "plan 진행"이라고 말씀해 주세요.
```

**중요**: 이 안내 후 사용자의 명시적 전환 요청이 있을 때까지 대기합니다. 사용자가 추가 피드백을 주면 Research를 계속 수정/보완합니다.

## 권장 모드 동작

- 체크포인트에서 사용자 응답을 기다리되, `skip`으로 건너뛸 수 있습니다
- 건너뛴 체크포인트는 산출물에 `skipped` 상태로 기록됩니다
- 모든 가정이 검증되지 않아도 다음 단계로 진행할 수 있지만, 경고 메시지를 표시합니다

## 사용할 Skills

이 명령어 수행 시 `research-methodology` skill을 로드하여 체계적인 조사 방법론을 적용합니다.

## 사용할 Agents

필요 시 다음 agent를 활용합니다:

- **verifier**: 체크포인트 관리 및 검증 요청
- **complexity-analyst**: 복잡성 분석 수행
